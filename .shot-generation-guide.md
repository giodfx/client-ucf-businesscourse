# Shot Generation Guide — Qwen Image Edit Best Practices

## What Works

### 5-Shot System v3 (position + zoom from avatar, no chain)
| Shot | Type | Zoom Level | Position | Base Image |
|------|------|-----------|----------|------------|
| 1 | Medium Shot | Waist up | LEFT (original) | COPY of avatar |
| 2 | Medium Shot | Waist up | RIGHT | Edit from avatar |
| 3 | Medium Close-up | Collarbone up | CENTER | Edit from avatar |
| 4 | Medium Close-up | Mid-chest up | RIGHT | Edit from avatar |
| 5 | Close Shot | Collarbone up (bigger) | CENTER | Edit from avatar |

### No Chain — All From Avatar (v3)
- **Shot 1** is a direct COPY of the source avatar (guaranteed correct outfit/background)
- **Shots 2-5** are ALL generated independently from the avatar (no chain = no drift)
- Every edit prompt includes "Keep the EXACT same clothing, hairstyle, jewelry, accessories"
- This prevents outfit drift that occurred with the old chain approach (Shot 1 → 2 → 3 → etc.)

### OLD Chain Approach (DEPRECATED — caused outfit drift)
- Shot 1 from avatar → Shot 2,3,5 from Shot 1 → Shot 4 from Shot 5
- Problem: each step in chain accumulated small changes (outfit, accessories, style)
- Problem: shots 1&2 were near-identical when using descriptive LEFT/RIGHT prompts

### Prompt Style: Edit Instructions vs Descriptions
- **WRONG**: "MEDIUM CLOSE-UP from chest up, character on the LEFT side" (descriptive — model ignores framing)
- **RIGHT**: "Change the framing to a medium close-up cropped at the collarbone. Show the upper chest, neck and head only." (edit instruction — model follows)
- Key: tell the model WHAT TO CHANGE, specify crop boundaries explicitly

### Zoom Control
- Qwen Image Edit CAN change zoom level, but subtly
- Use explicit crop boundary language: "cropped at the collarbone", "cropped at mid-chest"
- Words like "BIGGER" and "ZOOMED IN" alone are NOT enough
- Edit-instruction style ("Change the framing to...") works better than descriptive style
- Getting 3 distinct zoom levels (medium → medium close-up → close) is achievable

### Left/Right Positioning
- Works well with: "character on the LEFT/RIGHT side of the frame, more background visible on the other side"
- Combined with zoom creates 5 visually distinct shots

## What Does NOT Work

### Dynamic Elements in Backgrounds
- **Cars, vehicles** → InfinityTalk animates them (cars move backwards)
- **Roads, streets** → Implies motion, InfinityTalk may add movement
- **Waves, water** → Will animate
- **People walking** → Will animate
- Solution: Use static backgrounds (rooftop, building walls, skyline, architecture)

### Source Avatar Background Bleed
- If original avatar has cars/street, Qwen Image Edit preserves them ~50% of the time
- Even "no cars no road no street" in the prompt doesn't fully prevent bleed
- Adding "cars, vehicles, road, street" to negative prompt helps but isn't foolproof
- **Only reliable fix**: Chain approach (use Shot 1 with clean background as base for all other shots)

### Prompt Approaches That Failed
- "FULL BODY from knees up, character SMALL" → Subject stays same size
- "EXTREME CLOSE-UP, face fills 70% of frame" → Too extreme, ugly result
- "THREE-QUARTER PROFILE VIEW, TURNED 45 degrees" → Side views don't work well in InfinityTalk video
- "RIGHT hand touching chin thoughtfully" → Unnatural pose, user hated it
- "character positioned in the RIGHT THIRD of the frame" → Positioning words alone ignored
- Subtle framing differences ("waist up" vs "chest up" vs "hips up") → Model can't distinguish

### Video-Specific Issues
- Far/wide shots don't look good in InfinityTalk video (character too small)
- Side-positioned characters look unnatural in talking-head video
- "Static background" in InfinityTalk prompt does NOT prevent background animation
- InfinityTalk negative prompt for background motion has minimal effect

## Scene Description Rules (for all modules)

### DO Include
- Buildings, architecture (static)
- Skyline, city silhouettes (static)
- Trees, vegetation (mostly static)
- Sky, clouds, sunset (static enough)
- Rooftops, terraces, balconies

### DO NOT Include
- Cars, vehicles, traffic
- Roads, streets, highways, pavement
- Moving water, waves, rivers
- People walking, crowds
- Rides, moving machinery
- Any element that implies motion

## Technical Parameters
- **Denoise**: 1.0 (maximum — needed for framing changes)
- **CFG**: 1 (with Lightning LoRA)
- **Steps**: 4 (with Lightning LoRA)
- **Negative prompt**: Must include "cars, vehicles, road, street, pavement, traffic, highway, automobile"
- **Shot timeout**: 300000ms (5 min — some shots take 2+ minutes)
- **Anti-photorealism**: "Stylized painting, NOT a photograph, NOT photorealistic, NOT airbrushed"

## InfinityTalk Prompt
- Include "static background, still environment, only face and mouth moving"
- Negative: "moving background, camera motion, moving cars, background animation"
- Note: These help but cannot override dynamic elements in the source image
